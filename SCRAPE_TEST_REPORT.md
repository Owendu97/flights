# 携程价格抓取验证结果

**路由**：上海 (SHA) → 昆明 (KMG)
**抓取时间**：2026-07-08 10:22-10:35 (Asia/Shanghai)
**数据源**：携程 flights.ctrip.com — 搜索结果页"更多日期"区域
**方法**：Playwright 模拟真实浏览器，参数化 URL 中 depdate
**反爬情况**：9 次连续请求（跨 2 个月）全程未触发验证码 / IP 限制 / 反爬弹窗

---

## 核心发现（最重要的 3 条）

### 1. 携程"更多日期"日历直接暴露 7 天最低价

URL 模板：
```
https://flights.ctrip.com/online/list/oneway-sha-kmg?_=1&depdate=YYYY-MM-DD&cabin=Y_S_C_F&adult=1&child=0&infant=0
```

当 depdate = D 时，页面"更多日期"区域展示 D-3 到 D+3 共 7 天的最低价，标"低"的是该航线全月最低。

### 2. 修改 depdate 参数，日历跟随动态更新

- depdate=2026-08-10 → 日历显示 08-07 ~ 08-13
- depdate=2026-08-15 → 日历显示 08-12 ~ 08-18
- depdate=2026-09-10 → 日历显示 09-07 ~ 09-13（跨月正常）

### 3. 这就是用户要的核心数据

**用户的核心需求**：根据历史数据，告诉用户"提前 X 天购买的价格最低"。

**携程日历的数据语义**：
- depdate = D 时，08-09 周日的 ¥600 = "计划 08-09 出发，今天（2026-07-08）搜到的最低价"
- 也就是：**"今天到 08-09 还有 32 天，提前 32 天预订的当前价格"**
- 明天再抓一次 depdate=2026-08-10，08-09 那条数据就变成"提前 31 天"
- ...

**累积策略**：每天抓 depdate = 今天 ~ 今天+60 共 9 个 depdate（每个 depdate 覆盖前后 3 天），9 次请求就能覆盖"提前 0~60 天的最低价分布"。持续运行几个月 = 完整的历史时间序列。

## 可行性结论

### ✅ 完全可行

1. **数据获取**：携程"更多日期"日历直接暴露 7 天最低价，通过 depdate 参数可程序化抓取
2. **反爬现状**：9 次连续跨月请求 0 触发，短期内无风险
3. **数据语义**：直接对应用户需求"提前 N 天的最低价"
4. **技术栈**：Playwright + Python 即可，门槛低

### ⚠️ 需要注意的风险

1. **长期反爬升级风险**：每天 9 次 × N 个航线的规模，长期 1-3 个月可能触发携程风控
   - **应对**：每个 IP / session 控制 QPS（比如每航线 5 分钟一次）；准备住宅代理池作为备用
2. **数据是"展示价"不是"成交价"**：携程日历显示的是当前可订的最低价，会随库存变化
   - **应对**：连续抓取取均值，符合用户"均价参考"的需求；标注数据来源和"仅供参考"
3. **每个 depdate 只能覆盖 ±3 天**：要累积"提前 60 天的价格分布"需要 9 次 depdate 请求
   - **应对**：每次采集请求量固定且不大，scale 上没问题
4. **节假日价格波动剧烈**：中秋 ¥880 vs 9 月平日 ¥400，差异 2 倍以上
   - **应对**：MVP 必须按"节假日 vs 平日"分层建模，否则建议会误导用户

### ❌ 不确定的（需要继续验证）

1. **持续 7×24 抓取 1 个月后是否触发风控**：本次只测了短时高频，没测长期持续
2. **携程"价格日历"是否所有航线都有**：本次只测 SHA→KMG 一条航线，少数航线可能没日历
3. **数据是否稳定（同一天多次抓取结果一致）**：本次 9 次请求时间跨度内数据看起来稳定，但需要进一步验证
4. **反爬触发后的应对方案**：触发后能否解封、解封周期、是否需要换 IP 都没测过
