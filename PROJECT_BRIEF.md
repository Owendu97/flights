# 机票历史最低价参考小程序 — 项目档案

> 本文档汇总了项目前期的所有讨论、决策、技术验证结果，用于后续研发推进时快速调取上下文。
> 最后更新：2026-07-08

---

## 一、项目一句话定位

**基于携程价格日历数据，做一个微信小程序，告诉用户"这条航线、提前 X 天购买最可能买到最低价"。只展示预测参考，不做实时比价，不出票交易。**

---

## 二、产品边界

### ✅ 做什么
- 国内航线的历史最低价分位线（p10/p25/p50）
- 提前购买天数建议（基于历史均值/分位线）
- 同航线多航班价格趋势
- 节假日 / 周末 / 时段分桶建议
- Y 舱理论最低价参考（基于反推全价）
- 关键发现模块（"周六出发最便宜""晚班折扣最低"等）

### ❌ 不做什么（MVP 阶段）
- 实时比价 / 库存查询
- 出票 / 退改签（需要 IATA + 民航代理资质）
- 国际航线
- 团队票 / 协议价
- 燃油费 / 基建费实时跟踪（首页公告免责即可）
- 行李额度（V2 完善）
- 退改规则（V2 完善）

## 四、核心数据方案（已验证可行）

### 数据源
**携程 flights.ctrip.com 搜索结果页 "更多日期" 日历 + 当日航班列表**

### 抓取方法
- **工具**：Playwright（参数化浏览器，无头模式）
- **URL 模板**：
  ```
  https://flights.ctrip.com/online/list/oneway-{orig}-{dest}?_=1&depdate={YYYY-MM-DD}&cabin=Y_S_C_F&adult=1&child=0&infant=0
  ```
- **锚点策略**：depdate=D 时，"更多日期"区域展示 D-3 ~ D+3 共 7 天最低价

### 反爬状态
**✅ 11 次连续请求（跨 2 个月）0 触发验证码/IP 限制/反爬弹窗**

短期测试 OK。**未验证长期 7×24 抓取**（建议跑 2 周再判断）。

### 关键技术洞察

#### 1. 日历数据 = 用户核心需求
携程日历天然展示"7 天最低价"，**正好对应"提前 N 天购买的价格"**。

数据语义：
- 2026-08-22 抓到 8-25 ¥500 = "**2026-08-22 这一时刻**，航司/OTA 对 8-25 的定价"
- 不是历史最低价，是实时快照
- **持续抓取同一个未来日期在多个时间点的价格 = "提前 X 天购买的价格时间序列"**

#### 2. 修改 depdate 参数可累积历史
- 每天抓 depdate = 今天 ~ 今天+60 天（9 次请求覆盖前后 60 天窗口）
- 持续 6 个月 = 完整历史时间序列
- 单次请求拿 7 天，单航线 9 次请求覆盖 60 天窗口，性价比极高

#### 3. 折扣率反推全价 ⭐ 升级为 MVP 功能
携程页面已暴露折扣数据（"经济舱 2.9 折"、"85 折优惠券"），**直接反推 Y 舱全价**：

```
反推全价 = 当前价格 / 折扣率
例：¥620 / 0.29 = ¥2138（PEK-PVG 经济舱全价）
```

- 不需要爬航司官网全价
- 全价库可作为 V2 增强，但 MVP 已有基础
- 实际最低价 = Y × 最低折扣率（通常 0.15-0.20）
- 应用：价格底线参考 + 偏离度信号

## 六、数据模型（已按用户 10 要点更新）

```sql
CREATE TABLE price_calendar (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    route VARCHAR(20) NOT NULL,           -- 机场代码 'PEK-PVG', NOT '北京-上海'
    flight_date DATE NOT NULL,
    scrape_time TIMESTAMP NOT NULL,
    depdate_anchor DATE NOT NULL,
    days_before_departure INT NOT NULL,    -- flight_date - depdate_anchor
    price INT NULL,
    is_lowest BOOLEAN DEFAULT FALSE,
    raw_text TEXT,
    INDEX idx_route_date (route, flight_date, scrape_time),
    INDEX idx_days_before (route, flight_date, days_before_departure)
);

CREATE TABLE flight_snapshot (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    route VARCHAR(20) NOT NULL,           -- 机场代码
    flight_date DATE NOT NULL,
    scrape_time TIMESTAMP NOT NULL,
    flight_no VARCHAR(20) NOT NULL,        -- 主航班号（去共享）
    airline VARCHAR(50),
    aircraft VARCHAR(50),
    dep_airport VARCHAR(10),
    arr_airport VARCHAR(10),
    dep_time TIME,
    arr_time TIME,
    stops INT DEFAULT 0,                  -- 经停次数（不含中转）
    time_bucket VARCHAR(10),              -- 0-6/6-9/9-15/15-21/21-24
    price INT NOT NULL,
    discount_rate DECIMAL(4,2),            -- 0.85 = 85 折 / 0.29 = 2.9 折
    inferred_full_price DECIMAL(8,2),      -- 反推全价 = price / discount_rate
    is_main_flight BOOLEAN DEFAULT TRUE,
    INDEX idx_route_date_price (route, flight_date, price),
    INDEX idx_route_date_bucket (route, flight_date, time_bucket, price),
    INDEX idx_route_date_days (route, flight_date, days_before_departure)
);
```

## 八、数据洞察（基于实测）

### 5 时段分布（26 个航班）
```
6-9 (早班)：5 个 (19%)
9-15 (白天)：1 个 (4%)   ← 白天最少
15-21 (下午)：7 个 (27%)
21-24 (晚班)：13 个 (50%)  ← 晚班占一半！
0-6 (红眼)：0 个 (0%)
```

**晚班折扣率最低（1.7-2.5 折）**，MVP 应把"晚班推荐"作为重点输出。

### 折扣率反推全价样本
| 航班 | 当前价 | 折扣 | **反推全价** |
|------|--------|------|------------|
| MU5100 PEK-PVG | ¥700 | 3.3 折 | **¥2121** |
| MU5822 SHA-KMG | ¥600 | 1.8 折 | **¥3333** |
| MU5359 SHA-SZX | ¥400 | 1.8 折 | **¥2222** |
| HO1105 SHA-XMN | ¥450 | 2.8 折 | **¥1607** |

**Y 舱全价区间**：
- PEK-PVG：~¥2100
- SHA-XMN：~¥1600-2000
- SHA-KMG：~¥3300（距离远，**全价更贵但折扣率更低**）

### 航司定价策略
- **东航 MU**：覆盖所有时段，折扣率中等（2-2.5 折）
- **国航 CA**：覆盖 9-15 时段，折扣率较低（2.9 折）
- **吉祥 HO**：覆盖早班和晚班，折扣率激进（2.3-2.8 折）
- **上海航空 FM**：覆盖早班

## 九、技术栈建议

| 层 | 选型 | 理由 |
|---|------|------|
| 小程序 | 微信原生 或 uni-app | 看前端熟悉度 |
| 后端 | Python FastAPI | 异步好，数据处理生态成熟 |
| 数据库 | MySQL + Redis | 历史价数据量小，MySQL 完全够；Redis 缓存热点查询 |
| 爬虫 | Playwright | 携程 SPA 重 JS，requests 不够用 |
| 定时任务 | APScheduler 或 cron | 每天 1-2 次足够 |
| 部署 | 轻量云服务器 + cron | MVP 阶段成本极低 |

## 十、我的不确定清单（诚实清单）

按优先级排列，研发推进时需要逐步验证：

### 数据层
1. **长期反爬触发阈值**：跑 1-2 周才知道每天 9 次 × N 航线 × 30 天的规模是否安全
2. **携程日历是否有 CDN 缓存**：影响"提前 3 天的价格"实时性
3. **多 depdate 时间序列稳定性**：同一未来日期在不同时间点的数据是否合理波动
4. **小众航线的日历覆盖度**：只测了 6 条热门线，部分航线可能没日历
5. **共享航班识别规则集**：需要建库
6. **TFU（成都天府）等其他双机场**：只验证了 PKX

### 产品层
7. **微信小程序"出行"类目对纯展示类应用的具体资质要求**：需要查最新文档
8. **民航局对"机票价格预测/建议"类应用的监管态度**：这块一直在收紧
9. **数据是"展示价"还是"成交价"的偏差**：携程展示价会随库存变化，可能偏离实际成交价
10. **节假日波动对模型的冲击**：中秋 ¥880 vs 平日 ¥400 差 2 倍，必须分层建模

### 抓取层
11. **航班列表抓取的完整度**：本次抓到 1-4 个/航线，实际 5-10 个
12. **airline 字段错配**：CA 标 ""，春秋廉航错标成东航 —— 正则不全
13. **不同航线 DOM 结构差异**：可能需要为不同航线定制提取逻辑

## 十四、原始对话要点速查

### 关键讨论里程碑
1. **第一轮**：用户想做"实时机票资讯台"，我建议用 API（携程/飞猪爬虫不靠谱）
2. **第二轮（关键转折）**：用户澄清是基于历史数据的预测，不是实时比价 —— 大幅降低数据获取难度

### 我提出的关键修正
