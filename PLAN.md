# 机票历史最低价参考小程序 — MVP 规划

## 一、产品定义

**本质**：基于历史票价数据，告诉用户「这条航线，**提前多少天**买、**哪个时间窗口**最可能买到最低价」。

**关键边界**：
- **不是预测**，是参考 —— 不承诺「按这个买就是最低」
- **不是实时比价**，不展示当下价格 —— 避免和 OTA 正面竞争
- **只做国内、散客** —— 团队票/国际票逻辑完全不同，先不做

**两个核心输出**：
1. 这条航线近 N 个月的最低价分位线（p10/p25/p50）
2. 起飞前多少天购买最划算（基于历史成交价的时间分布）

---

## 二、思路梳理与盲点

### 你已经做对的事

- ✅ 把产品定位成「历史参考」而非「实时预测」 —— 这绕开了 80% 的数据获取坑
- ✅ 接受「均价参考」而不是精确预测 —— 工程复杂度大幅下降
- ✅ 区分四大航司和小众航线 —— 体现了代理经验，知道数据天然分层
- ✅ 微信小程序形态 —— 国内用户触达成本最低

### 你可能没充分意识到的几个点

#### 1. 数据冷启动是真正的核心难题
历史趋势分析**前提是有历史数据**。冷启动期（0 数据）你什么都给不出来。

- 选项 A：爬公开数据攒 6-12 个月再做（推荐，先跑起来）
- 选项 B：你自己/老同事手里有没有代理侧脱敏后的历史成交价数据？**这是金矿**，任何公开 API 都给不出来
- 选项 C：第三方价格历史数据集（基本没有可用的）

#### 2. 「历史最低价」的来源问题
公开渠道能稳定拿到的历史价格很少：

| 来源 | 能拿到的 | 难度 |
|------|---------|------|
| 携程「低价日历」 | 单条航线未来 1-2 个月的趋势 | 中（要验证反爬可行性） |
| 飞猪/去哪儿价格日历 | 类似 | 中（反爬更严） |
| 航司官网历史价格 | 几乎不留痕 | 高 |
| Hopper / Kayak 价格趋势 | 国内航线覆盖差 | - |
| Google Flights 历史价 API | 已关停 | - |
| **你自己的代理成交数据** | 最全最真实 | **0（要看你能不能拿到）** |

**关键判断**：携程低价日历是当前最现实的爬取目标，但需要先验证。

#### 3. 影响价格的因素至少 6 个，建模前要想清楚分层
- 节假日（春节/十一/寒暑假）vs 平日 —— 规律完全不同
- 起飞时间（早班 vs 晚班）—— 同一天能差 30%
- 工作日 vs 周末 —— 周末一般贵 10-20%
- 是否热门航线（京沪 vs 三四线城市互飞）
- 提前天数 —— 这是核心预测维度
- 是否廉航/共享航班 —— 价格体系完全不同

**MVP 先做时间维度 + 节假日分层**，其他维度降级为提示文本。

#### 4. 预测不是 100% 准，必须加免责
- 小程序里必须明确「仅供参考」「历史不代表未来」
- 避免用户买了高价找你维权 —— 你可以做工具但不承担结果
- 微信对涉及金钱建议类小程序有合规要求

#### 5. 微信小程序的「出行」类目资质
- 单纯展示价格 + 推荐 → 不需要 IATA / 民航代理资质
- 但如果未来加「帮用户下单」或「跳转购买」 → 需要资质
- 建议 MVP 阶段**严格只做展示推荐**，跳转走用户自己手动复制航司官网或 OTA 链接

#### 6. 模型不需要复杂
- 移动平均 + 分位线 + 按月/按周季节性分解就够了
- 不要碰 LSTM / 时序模型 —— 这种数据量小、噪声大，复杂模型反而过拟合
- Python `statsmodels` 的 seasonal_decompose + 简单分位数统计，足够支撑 MVP

## 三、MVP 三阶段路径

### 阶段 1：数据采集验证（1-2 周）
**目标**：验证核心数据能不能稳定拿到

- 选定 10-20 条热门航线（北京-上海、北京-广州、上海-深圳 等）
- 爬携程「低价日历」（**先小规模验证反爬可行性**）
- 同时调研：飞猪/去哪儿/航旅纵横/Google Flights 是否有可替代来源
- 关键里程碑：能在本地稳定存下某条航线连续 7 天的价格日历数据

**如果这一关过不去，整个项目需要重新评估数据来源策略。**

### 阶段 2：历史数据积累 + 简单统计（3-6 个月）
**目标**：攒出能用的历史数据集

- 后端定时爬（每天 1-2 次），存 MySQL/SQLite
- 不做任何展示，纯粹攒数据
- 同时做：
  - 数据清洗（剔除异常值，比如明显是 bug 的 9 元机票）
  - 按「提前天数 × 航线 × 是否节假日」分桶
  - 计算每条航线的 p10/p25/p50 历史价

**这是产品的核心资产，不可跳过。**

### 阶段 3：MVP 小程序上线（数据够用时启动）
**目标**：让用户能查询到参考建议

- 微信小程序，原生或 uni-app
- 后端 Python FastAPI / Node.js 都行
- 功能：
  - 输入：出发地、目的地、计划出行月份
  - 输出：建议提前 X 天购买 + 该航线的历史最低价分位线
  - 小众航线：宽泛建议 + 标注「数据不足，建议提前 21 天关注」
- 明确免责声明

## 四、技术栈建议

| 层 | 选型 | 理由 |
|---|------|------|
| 小程序 | 微信原生 或 uni-app | 看你的前端熟悉度 |
| 后端 | Python FastAPI | 异步好，数据处理生态成熟 |
| 数据库 | MySQL（航线/历史价） + Redis（缓存） | 历史价数据量小，MySQL 完全够 |
| 爬虫 | Playwright（首选，携程 SPA 重 JS） | requests 不够用 |
| 定时任务 | APScheduler 或简单 cron | 每天 1-2 次足够 |
| 部署 | 云函数 + 轻量云服务器 | MVP 阶段成本极低 |

## 五、不确定的地方（诚实清单）

1. **携程「低价日历」的反爬强度**：我倾向于可破，但需要你或我跑一次实测才能确认
2. **飞猪/去哪儿是否有更友好的价格日历接口**：需要验证
3. **国内航司官网是否提供任何价格历史查询**：国航/东航官网历史价展示我印象里没有，不确定现在是否变化
4. **微信小程序「出行」类目对纯展示类应用的具体资质要求**：需要查最新文档
5. **民航局对「机票价格预测/建议」类应用的监管态度**：这块一直在变，我没有 2026 年的最新依据
6. **你的代理背景能不能接触到脱敏后的历史成交数据**：这是最关键的变量，你比我清楚
7. **携程开放平台/航旅纵横开放平台对「历史价格分析」类应用的态度**：上一轮我列过开放平台不开放给个人，但历史数据这个细分场景可能不同
8. **国内航司 NDC 是否提供历史价回溯**：一般不提供，但我没 100% 把握

## 六、关键决策点（需要你来定）

1. **你/老同事手里能不能拿到哪怕小规模的脱敏历史成交数据？**
   - 能 → 数据冷启动快 6 个月以上，项目可行性大幅提升
   - 不能 → 完全靠爬公开数据，需要 6-12 个月数据积累期

2. **携程低价日历**这个数据来源，你想先让我帮你做技术验证吗？
   - 验证一次大概 2-3 天，能告诉你这个项目能不能跑

3. **小众航线的「宽泛建议」具体给到什么粒度？**
   - 选项 A：「这类航线建议提前 21-30 天购买」（非常宽泛）
   - 选项 B：「这条航线数据不足，参考临近热门航线」（部分基于相似度）
   - 选项 C：直接不展示小众航线
   - 我倾向 A+B 结合，但需要你定基调

## 七、下一步建议

**立刻可以做的**：选定 5-10 条最热门航线，跑一次「携程低价日历」爬取可行性验证，2-3 天能出结果。这一步如果过不去，所有规划要重做；如果过得去，剩下就是攒数据的时间问题。
