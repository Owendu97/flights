## 三、回应用户的 4 个补充

### 1. 同航线不同时刻不同航班 ✅ 验证可行

**结论**：携程日历 = 当日所有航班最低价。日历已经覆盖了用户的核心需求。

**生产方案**：
- 主数据：日历数据（"提前 X 天购买"的最低价）
- 副数据：当日航班列表（用于异常检测和细节展示）
- **不需要**按航班粒度建索引（数据爆炸）

### 2. 换季空价格 ⚠️ 实证出现

**沪深 2026-08-13 价格为空** —— 真实存在。

**应对方案**：
```sql
CREATE TABLE price_calendar (
    route VARCHAR(20),
    flight_date DATE,
    scrape_time TIMESTAMP,
    price INT NULL,           -- 允许 NULL
    status VARCHAR(20),       -- 'priced' / 'pending_pricing' / 'sold_out'
    raw_data JSON
);
```
- 价格 NULL 时不参与分位线计算
- 标注 `status='pending_pricing'` 或 `'sold_out'`
- 跨天后重新抓取，看是否恢复

### 3. 日历数据的语义 ⚠️ 关键澄清

**用户问题**：8-25 的机票在 8-22 看到的 ¥500，是不是历史最低价？

**答案**：**不是历史最低价，是 8-22 这一时刻的"展示价"**。

**这正好对应用户需求**：
- 持续抓取同一个未来日期在多个时间点的价格 = "提前 X 天购买的价格时间序列"
- 携程日历数据本质上是**面向未来的实时快照**，会随时间动态变化
- **每天抓 depdate = 今天 ~ 今天+60 天**，积累 6 个月 = 完整的历史序列

### 4. Y 舱理论最低价 ⭐ V2 增强

**思路**：理论最低价 = Y 舱全价 × 最低折扣率（通常 30-40%）

**价值**：
- 给旅客"价格底线参考"
- 携程展示价 vs 理论最低价 → 偏离度 → 价格趋势信号

**落地难度**：
- Y 舱全价需要民航局/航司数据源（不开放 API）
- 折扣舱位体系各航司差异大，需要建库

**建议**：
- **MVP 不做**（先把日历数据落地）
- **V2 启动**：先手动建 10 条热门航线的 Y 舱全价库
- **V3 自动**：爬航司官网标准运价展示

## 四、用户思路遗漏项（我的补充）

1. **周末效应** —— 周五、周日的票通常贵 10-20%，日历里能看出来（比如京沪 8-07 周五 ¥550 vs 其他日 ¥350）。MVP 应该按 weekday 分桶
2. **早晚班价格差异** —— 沪深 FM9275 早班 ¥926 vs MU5359 晚班 ¥390，差 2.4 倍。**这是航班时刻溢价，不能用单一最低价掩盖**
3. **起飞时间窗口** —— 同一天不同时刻价格差异巨大，是否要给用户分"早班推荐"/"晚班推荐"两种？
4. **燃油费/机场建设费** —— 通常 ¥50-100，会随政策变化。**展示给用户的应该是含税总价**
5. **退改条款** —— 最低价往往是不可退改的。**展示建议时必须标注"是否可退改"**
6. **航司直销 vs OTA 价格差** —— 同一航班，官网 vs 携程可能差 5-15%。建议要不要区分"航司最低"和"OTA 最低"？
7. **小程序 UI 的关键问题**：用户看到"建议 8-22 买" → 实际上要怎么操作？**建议提示用户回到携程或航司官网搜具体航班**，还是直接 deep-link？

## 五、我的不确定（诚实清单）

1. **08-13 沪深空价格的具体原因** —— 是 pending_pricing（航司未定价）、sold_out（售完）、还是 hide（携程主动隐藏）？需要看携程的内部状态字段
2. **多次抓取同一 depdate 的数据稳定性** —— 本次每个 depdate 只抓 1 次，没测"同一时刻 5 次抓取结果是否完全一致"
3. **更多日期日历的缓存策略** —— 携程是否对日历数据做了 CDN 缓存？如果有，"提前 3 天的价格"实时性可能降低
4. **携程日历的"低"标签定义** —— 是基于全月对比、本周对比、还是 30 天对比？标签定义会影响数据解读
5. **当日航班列表的完整提取** —— 本次 DOM 抓取只命中部分，需要优化选择器
6. **周末/早班晚班差异** —— 是否需要按这些维度建立独立分位线？
7. **反爬的长期触发阈值** —— 跑 1 周才知道

## 六、调整后的产品方案（基于本轮验证）

### MVP 数据模型

```sql
-- 主表：日历数据
CREATE TABLE price_calendar (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    route VARCHAR(20) NOT NULL,           -- 'SHA-KMG', 'PEK-SHA', 'SHA-SZX'
    flight_date DATE NOT NULL,             -- 实际起飞日期
    scrape_time TIMESTAMP NOT NULL,        -- 抓取时间
    depdate_anchor DATE NOT NULL,          -- 抓取时使用的 URL 锚点
    days_before_departure INT NOT NULL,    -- 提前天数 = flight_date - depdate_anchor
    price INT NULL,                        -- 价格，NULL 表示未定价或售完
    status VARCHAR(20) DEFAULT 'priced',   -- priced / pending_pricing / sold_out
    is_lowest BOOLEAN DEFAULT FALSE,       -- 是否标"低"
    raw_text TEXT,                         -- 原始抓取文本
    INDEX idx_route_date (route, flight_date, scrape_time),
    INDEX idx_days_before (route, flight_date, days_before_departure)
);

-- 副表：当日航班列表（用于异常检测 + 细节展示）
CREATE TABLE flight_snapshot (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    route VARCHAR(20) NOT NULL,
    flight_date DATE NOT NULL,
    scrape_time TIMESTAMP NOT NULL,
    flight_no VARCHAR(20) NOT NULL,
    airline VARCHAR(50),
    price INT NOT NULL,
    dep_time TIME,
    arr_time TIME,
    discount_rate DECIMAL(4,2),            -- 8.5 折 = 0.85
    is_lowest BOOLEAN DEFAULT FALSE,
    INDEX idx_route_date_price (route, flight_date, price)
);
```

### MVP 阶段目标（3 阶段）

#### 阶段 1：多航线数据采集（1-2 周）
- 选定 20-30 条热门航线
- 每天抓 depdate = 今天 ~ 今天+60 天（每个 depdate 覆盖前后 3 天）
- 单航线 9 次请求 × 30 条线 = 每天 270 次请求
- 频率控制：每 5 秒 1 次 = 全量 22 分钟完成
- **重点观察**：是否触发反爬、哪些航线无日历数据

#### 阶段 2：换季处理 + 数据清洗（持续）
- 检测 `price IS NULL` 情况，分类 pending_pricing / sold_out
- 异常值过滤（< ¥100 或 > 全价 × 1.5 的剔除）
- 按 weekday / 节假日 / 航线分别建立分位线

#### 阶段 3：MVP 小程序上线
- 输入：出发地、目的地、计划出行月份
- 输出：
  - 建议提前 X 天购买
  - 该航线近 6 个月最低价分位线（p10/p25/p50）
  - 当日航班列表（按价格排序）
- 严格免责声明：「仅供参考，历史不代表未来」

### V2 增强（3-6 个月后）
- Y 舱理论最低价模型
- 早晚班推荐
- 周几推荐
- 燃油费/退改条款展示
